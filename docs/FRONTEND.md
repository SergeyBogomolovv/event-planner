# Техническое описание фронтенда

Документ описывает фронтенд-часть веб-приложения для организации мероприятий. Здесь фиксируются страницы, основные состояния интерфейса, пользовательские потоки и связь с backend API на концептуальном уровне, без выбора конкретного фреймворка, UI-библиотеки или CSS-подхода.

## 1. Общий подход

Фронтенд отвечает за:

- публичные страницы для гостя;
- регистрацию и вход пользователя;
- личный кабинет авторизованного пользователя;
- создание и управление мероприятиями;
- просмотр приглашений;
- принятие и отклонение участия;
- просмотр мероприятий, где пользователь участвует;
- управление участниками для организатора;
- административную панель;
- отображение уведомлений и статусов.

Приложение должно учитывать, что все мероприятия закрытые. Страница конкретного мероприятия доступна только организатору, приглашённым пользователям, участникам и администратору.

## 2. Пользовательские зоны

### 2.1. Public Area

Доступна гостям и авторизованным пользователям.

Включает:

- главную страницу;
- страницу регистрации;
- страницу входа.

### 2.2. App Area

Доступна только авторизованным пользователям.

Включает:

- личный кабинет;
- мои мероприятия;
- создание мероприятия;
- редактирование мероприятия;
- приглашения;
- я участвую;
- страницу мероприятия;
- уведомления;
- профиль пользователя.

### 2.3. Admin Area

Доступна только пользователям с ролью `admin`.

Включает:

- обзор статистики;
- управление пользователями;
- управление мероприятиями.

## 3. Маршруты

Базовая структура маршрутов:

| Маршрут | Страница | Доступ |
| --- | --- | --- |
| `/` | Главная страница | Все |
| `/register` | Регистрация | Гость |
| `/login` | Вход | Гость |
| `/dashboard` | Личный кабинет | Пользователь |
| `/events/my` | Мои мероприятия | Пользователь |
| `/events/new` | Создание мероприятия | Пользователь |
| `/events/participating` | Я участвую | Пользователь |
| `/events/:eventId` | Страница мероприятия | Организатор, приглашённый, участник, администратор |
| `/events/:eventId/edit` | Редактирование мероприятия | Организатор |
| `/events/:eventId/participants` | Участники мероприятия | Организатор, приглашённый, участник |
| `/events/:eventId/participants/manage` | Управление участниками | Организатор |
| `/invitations` | Приглашения | Пользователь |
| `/notifications` | Уведомления | Пользователь |
| `/profile` | Профиль | Пользователь |
| `/admin` | Админ-панель | Администратор |
| `/admin/users` | Пользователи | Администратор |
| `/admin/events` | Мероприятия | Администратор |

Если пользователь открывает закрытый маршрут без авторизации, его нужно перенаправить на страницу входа.

Если пользователь авторизован, но не имеет прав на страницу, нужно показывать состояние "нет доступа" или перенаправлять на личный кабинет.

## 4. Основные страницы

### 4.1. Главная страница

Назначение:

- объяснить назначение сервиса;
- дать переход к регистрации и входу.

Основные блоки:

- краткое описание сервиса;
- список ключевых возможностей;
- кнопки регистрации и входа.

Данные с API не требуются.

### 4.2. Регистрация

Назначение:

- создать новый аккаунт пользователя.

Основные поля:

- имя;
- email;
- пароль;
- подтверждение пароля.

API:

- `POST /api/v1/auth/register`.

Состояния:

- форма не заполнена;
- отправка данных;
- ошибки валидации;
- email уже занят;
- успешная регистрация.

После успешной регистрации пользователь может быть перенаправлен на вход или сразу в личный кабинет, в зависимости от выбранной backend-логики.

### 4.3. Вход

Назначение:

- авторизовать пользователя.

Основные поля:

- email;
- пароль.

API:

- `POST /api/v1/auth/login`;
- `GET /api/v1/auth/me`.

Состояния:

- форма не заполнена;
- отправка данных;
- неверные данные;
- аккаунт заблокирован;
- успешный вход.

После успешного входа пользователь попадает в личный кабинет.

### 4.4. Личный кабинет

Назначение:

- показать пользователю сводку по его активности.

Основные блоки:

- ближайшие мероприятия;
- созданные мероприятия;
- мероприятия, где пользователь участвует;
- новые приглашения;
- последние уведомления;
- быстрые действия.

API:

- `GET /api/v1/dashboard`;
- при необходимости `GET /api/v1/notifications`.

Действия:

- создать мероприятие;
- открыть мероприятие;
- открыть приглашения;
- перейти к своим мероприятиям;
- перейти к мероприятиям участия.

### 4.5. Мои мероприятия

Назначение:

- показать мероприятия, где текущий пользователь является организатором.

API:

- `GET /api/v1/events/my`.

Основные элементы:

- список мероприятий;
- статус мероприятия;
- дата и время;
- количество подтверждённых участников;
- быстрые действия.

Действия:

- открыть мероприятие;
- создать мероприятие;
- редактировать мероприятие;
- опубликовать черновик;
- отменить мероприятие;
- завершить мероприятие;
- удалить мероприятие.

### 4.6. Создание мероприятия

Назначение:

- создать новое мероприятие в статусе `draft`.

Основные поля:

- название;
- описание;
- дата и время начала;
- дата и время окончания, опционально;
- место проведения;
- формат;
- лимит участников.

API:

- `POST /api/v1/events`.

После успешного создания пользователь переходит на страницу мероприятия или в список "Мои мероприятия".

### 4.7. Редактирование мероприятия

Назначение:

- изменить данные существующего мероприятия.

API:

- `GET /api/v1/events/{eventId}`;
- `PATCH /api/v1/events/{eventId}`.

Доступ:

- только организатор.

Состояния:

- загрузка данных;
- сохранение;
- ошибки валидации;
- мероприятие уже завершено;
- нет доступа.

### 4.8. Страница мероприятия

Назначение:

- показать полную информацию о мероприятии и доступные действия.

API:

- `GET /api/v1/events/{eventId}`;
- `GET /api/v1/events/{eventId}/participants`.
- для действий используются отдельные методы публикации, отмены, завершения, удаления и участия.

Основные блоки:

- название;
- описание;
- дата и время;
- место;
- формат;
- статус мероприятия;
- организатор;
- список подтверждённых участников;
- роль текущего пользователя относительно мероприятия;
- доступные действия.

Действия для организатора:

- редактировать;
- опубликовать;
- отменить;
- завершить;
- удалить;
- перейти к управлению участниками.

Действия для приглашённого пользователя:

- принять приглашение;
- отклонить приглашение.

Действия для участника:

- отказаться от участия.

Действия для администратора:

- просмотреть мероприятие;
- удалить мероприятие через административное действие.

Связь действий с API:

| Действие | API |
| --- | --- |
| Опубликовать | `POST /api/v1/events/{eventId}/publish` |
| Отменить | `POST /api/v1/events/{eventId}/cancel` |
| Завершить | `POST /api/v1/events/{eventId}/complete` |
| Удалить | `DELETE /api/v1/events/{eventId}` |
| Принять приглашение | `POST /api/v1/events/{eventId}/participants/{userId}/accept` |
| Отклонить приглашение | `POST /api/v1/events/{eventId}/participants/{userId}/decline` |
| Отказаться от участия | `POST /api/v1/events/{eventId}/participants/{userId}/leave` |

### 4.9. Приглашения

Назначение:

- показать мероприятия, на которые текущий пользователь приглашён.

API:

- `GET /api/v1/invitations`.
- `POST /api/v1/events/{eventId}/participants/{userId}/accept`;
- `POST /api/v1/events/{eventId}/participants/{userId}/decline`.

Основные элементы:

- название мероприятия;
- организатор;
- дата и время;
- статус мероприятия;
- действия принятия и отклонения.

Действия:

- открыть мероприятие;
- принять приглашение;
- отклонить приглашение.

При принятии или отклонении приглашения список должен обновиться.

### 4.10. Я участвую

Назначение:

- показать мероприятия, где пользователь подтвердил участие.

API:

- `GET /api/v1/events/participating`.
- `POST /api/v1/events/{eventId}/participants/{userId}/leave`.

Основные элементы:

- список мероприятий;
- дата и время;
- место;
- статус мероприятия;
- организатор.

Действия:

- открыть мероприятие;
- отказаться от участия.

### 4.11. Участники мероприятия

Назначение:

- показать пользователям, имеющим доступ к мероприятию, организатора и подтверждённых участников.

API:

- `GET /api/v1/events/{eventId}/participants`.

Основные элементы:

- организатор;
- подтверждённые участники;
- количество подтверждённых участников;
- лимит участников, если задан.

Обычные участники и приглашённые пользователи не видят пользователей со статусами `declined` и `removed`.

### 4.12. Управление участниками

Назначение:

- дать организатору управление приглашениями и участниками.

API:

- `GET /api/v1/events/{eventId}/participants/manage`;
- `GET /api/v1/users/search`;
- `POST /api/v1/events/{eventId}/participants`;
- `DELETE /api/v1/events/{eventId}/participants/{userId}`.

Основные элементы:

- поиск зарегистрированного пользователя;
- список приглашённых;
- список подтвердивших участие;
- список отказавшихся;
- список удалённых, если нужен для контроля;
- счётчик подтверждённых участников;
- лимит участников.

Действия:

- пригласить пользователя;
- повторно пригласить пользователя, если это разрешено backend-правилами;
- удалить участника или приглашённого;
- открыть профиль участника в базовом виде, если такая страница будет реализована.

### 4.13. Уведомления

Назначение:

- показать внутренние уведомления пользователя.

API:

- `GET /api/v1/notifications`;
- `PATCH /api/v1/notifications/{notificationId}/read`.

Основные элементы:

- список уведомлений;
- отметка о прочтении;
- ссылка на связанное мероприятие.

### 4.14. Профиль

Назначение:

- показать и редактировать данные текущего пользователя.

API:

- `GET /api/v1/users/me`;
- `PATCH /api/v1/users/me`.

Основные поля:

- имя;
- email.

На первом этапе смену пароля можно вынести в отдельный сценарий или отложить.

### 4.15. Админ-панель

Назначение:

- дать администратору обзор системы и инструменты управления.

API:

- `GET /api/v1/admin/stats`;
- `GET /api/v1/admin/users`;
- `GET /api/v1/admin/events`.

Основные блоки:

- количество пользователей;
- количество мероприятий;
- количество активных мероприятий;
- последние созданные мероприятия;
- быстрые переходы к пользователям и мероприятиям.

### 4.16. Управление пользователями

Назначение:

- дать администратору возможность просматривать и блокировать пользователей.

API:

- `GET /api/v1/admin/users`;
- `PATCH /api/v1/admin/users/{userId}/block`;
- `PATCH /api/v1/admin/users/{userId}/unblock`.

Основные элементы:

- таблица пользователей;
- роль;
- статус аккаунта;
- дата регистрации;
- действия блокировки и разблокировки.

### 4.17. Управление мероприятиями

Назначение:

- дать администратору возможность просматривать и удалять некорректные мероприятия.

API:

- `GET /api/v1/admin/events`;
- `DELETE /api/v1/events/{eventId}`.

Основные элементы:

- таблица мероприятий;
- организатор;
- статус;
- дата проведения;
- количество участников;
- действие удаления.

## 5. Общие компоненты

Компоненты стоит проектировать вокруг повторяющихся задач, а не вокруг конкретных страниц.

Базовые компоненты:

- layout приложения;
- верхняя навигация;
- боковая навигация для закрытой части;
- форма входа;
- форма регистрации;
- форма мероприятия;
- карточка мероприятия;
- список мероприятий;
- бейдж статуса мероприятия;
- бейдж статуса участия;
- список участников;
- строка участника;
- модальное подтверждение действия;
- поиск пользователя;
- список уведомлений;
- состояние загрузки;
- состояние ошибки;
- пустое состояние списка.

## 6. Клиентские состояния

### 6.1. Auth State

Хранит:

- текущего пользователя;
- факт авторизации;
- глобальную роль;
- статус аккаунта.

Используется для:

- защиты маршрутов;
- показа навигации;
- определения доступа к административной зоне.

### 6.2. Event Relation State

Для страницы мероприятия фронтенд должен получать или вычислять отношение текущего пользователя к событию:

- организатор;
- приглашённый;
- участник;
- администратор;
- нет доступа.

Лучше, если backend будет возвращать эти флаги вместе с мероприятием и списком доступных действий.

### 6.3. UI State

Для основных действий нужны состояния:

- загрузка;
- отправка формы;
- ошибка;
- успешное действие;
- пустой список;
- подтверждение опасного действия.

Опасные действия:

- отмена мероприятия;
- завершение мероприятия;
- удаление мероприятия;
- удаление участника;
- блокировка пользователя;
- удаление мероприятия администратором.

## 7. Навигация

Для гостя:

- главная;
- регистрация;
- вход.

Для пользователя:

- личный кабинет;
- мои мероприятия;
- приглашения;
- я участвую;
- уведомления;
- профиль.

Для организатора конкретного мероприятия дополнительные действия показываются внутри страницы мероприятия:

- редактировать;
- управление участниками;
- опубликовать;
- отменить;
- завершить;
- удалить.

Для администратора:

- админ-панель;
- пользователи;
- мероприятия.

Административная навигация должна отображаться только при `role = admin`.

## 8. Формы и валидация

Фронтенд выполняет базовую валидацию до отправки:

- обязательные поля;
- корректный email;
- минимальная длина пароля;
- дата начала мероприятия;
- дата окончания позже даты начала;
- лимит участников не меньше нуля;
- нельзя отправить пустое приглашение.

Backend всё равно должен повторно проверять все правила.

## 9. Ошибки

Типовые ошибки, которые нужно обработать:

- пользователь не авторизован;
- нет доступа;
- аккаунт заблокирован;
- мероприятие не найдено;
- мероприятие уже отменено или завершено;
- приглашение уже принято или отклонено;
- лимит участников достигнут;
- пользователь уже приглашён;
- пользователь не найден при поиске;
- ошибка отправки формы;
- ошибка загрузки списка.

Ошибки должны быть понятны пользователю и не раскрывать лишние технические детали.

## 10. Связь страниц с API

| Страница | Основные API |
| --- | --- |
| Регистрация | `POST /auth/register` |
| Вход | `POST /auth/login`, `GET /auth/me` |
| Личный кабинет | `GET /dashboard` |
| Мои мероприятия | `GET /events/my` |
| Создание мероприятия | `POST /events` |
| Редактирование мероприятия | `GET /events/{eventId}`, `PATCH /events/{eventId}` |
| Страница мероприятия | `GET /events/{eventId}`, `GET /events/{eventId}/participants`, `POST /events/{eventId}/publish`, `POST /events/{eventId}/cancel`, `POST /events/{eventId}/complete`, `DELETE /events/{eventId}` |
| Приглашения | `GET /invitations`, `POST /events/{eventId}/participants/{userId}/accept`, `POST /events/{eventId}/participants/{userId}/decline` |
| Я участвую | `GET /events/participating`, `POST /events/{eventId}/participants/{userId}/leave` |
| Управление участниками | `GET /events/{eventId}/participants/manage`, `GET /users/search`, `POST /events/{eventId}/participants`, `DELETE /events/{eventId}/participants/{userId}` |
| Уведомления | `GET /notifications`, `PATCH /notifications/{notificationId}/read` |
| Профиль | `GET /users/me`, `PATCH /users/me` |
| Админ-панель | `GET /admin/stats` |
| Админ пользователи | `GET /admin/users`, `PATCH /admin/users/{userId}/block`, `PATCH /admin/users/{userId}/unblock` |
| Админ мероприятия | `GET /admin/events`, `DELETE /events/{eventId}` |

В таблице указан путь без префикса `/api/v1`.

## 11. Что можно отложить

Для первой версии можно не реализовывать:

- календарный вид мероприятий;
- drag-and-drop планирование;
- публичный каталог мероприятий;
- страницы публичных профилей пользователей;
- расширенные настройки уведомлений;
- загрузку аватаров и изображений мероприятий;
- чат или комментарии внутри мероприятия;
- сложные фильтры и сохранённые представления;
- многошаговый мастер создания мероприятия.
