# Техническое описание бэкенда и базы данных

Документ описывает базовую техническую модель для реализации веб-приложения организации мероприятий. Здесь фиксируются основные сущности, связи, сервисы и API-контракты на концептуальном уровне, без выбора конкретных библиотек, ORM, SQL-диалекта или фреймворка.

## 1. Общий подход

Бэкенд отвечает за:

- регистрацию и аутентификацию пользователей;
- хранение профилей пользователей;
- управление мероприятиями;
- управление приглашениями и участниками;
- проверку прав доступа;
- административные операции;
- внутренние и email-уведомления;
- выдачу данных для личного кабинета и основных разделов приложения.

На первом этапе систему можно реализовывать как единое backend-приложение с логическим разделением на сервисы. Физически разбивать проект на микросервисы пока не требуется: домен небольшой, а основные сценарии сильно связаны между собой.

Принятые продуктовые ограничения для первой версии:

- приглашать можно только зарегистрированных пользователей;
- все мероприятия доступны только организатору, приглашённым пользователям, участникам и администратору;
- публичного каталога мероприятий нет;
- организатор должен отображаться в списке участников для других пользователей;
- участники могут видеть общий список участников мероприятия;
- email-уведомления входят в первую версию;
- отдельная история изменения статусов участия не хранится.

## 2. Логические сервисы

### 2.1. Auth Service

Отвечает за регистрацию, вход, выход и проверку текущего пользователя.

Зоны ответственности:

- создание аккаунта;
- проверка email и пароля при входе;
- выдача и обновление сессии или токена;
- проверка заблокированного статуса пользователя;
- получение данных текущего пользователя.

### 2.2. User Service

Отвечает за пользовательские данные и административное управление пользователями.

Зоны ответственности:

- просмотр и редактирование профиля;
- поиск пользователей для приглашения;
- список пользователей для администратора;
- блокировка и разблокировка пользователей;
- хранение глобальной роли пользователя.

### 2.3. Event Service

Отвечает за жизненный цикл мероприятий.

Зоны ответственности:

- создание мероприятия;
- редактирование мероприятия;
- публикация мероприятия;
- отмена мероприятия;
- завершение мероприятия;
- удаление мероприятия;
- получение списков мероприятий;
- проверка прав организатора.

### 2.4. Participation Service

Отвечает за приглашения и участие пользователей в мероприятиях.

Зоны ответственности:

- приглашение пользователя на мероприятие;
- просмотр приглашений;
- принятие приглашения;
- отклонение приглашения;
- отказ от участия;
- удаление участника организатором;
- подсчёт подтверждённых участников;
- проверка лимита участников.

### 2.5. Admin Service

Отвечает за операции администратора.

Зоны ответственности:

- просмотр всех пользователей;
- блокировка и разблокировка пользователей;
- просмотр всех мероприятий;
- удаление некорректных мероприятий;
- получение простой статистики системы.

### 2.6. Notification Service

Отвечает за внутренние уведомления и отправку email-уведомлений.

Зоны ответственности:

- создание внутренних уведомлений;
- уведомление пользователя о приглашении;
- уведомление участников об отмене или изменении мероприятия;
- отправка email при приглашении, изменении мероприятия и отмене мероприятия;
- хранение статуса отправки email.

На первом этапе это внутренний модуль основного backend-приложения. Email отправляется через BullMQ-задачи, а обработчик очереди живёт внутри NestJS backend-приложения. Отдельный worker-сервис и отдельный Docker image для worker не создаются.

## 3. Основные сущности базы данных

### 3.1. users

Хранит зарегистрированных пользователей.

Поля:

| Поле          | Назначение                            |
| ------------- | ------------------------------------- |
| id            | Уникальный идентификатор пользователя |
| email         | Email пользователя, уникальный        |
| password_hash | Хэш пароля                            |
| name          | Отображаемое имя                      |
| role          | Глобальная роль: `user`, `admin`      |
| status        | Статус аккаунта: `active`, `blocked`  |
| created_at    | Дата создания                         |
| updated_at    | Дата обновления                       |

Примечания:

- роли `organizer`, `invited` и `participant` не хранятся в `users`, потому что они зависят от конкретного мероприятия;
- администратор является глобальной ролью.

### 3.2. events

Хранит мероприятия.

Поля:

| Поле              | Назначение                                          |
| ----------------- | --------------------------------------------------- |
| id                | Уникальный идентификатор мероприятия                |
| organizer_id      | Пользователь, который создал мероприятие            |
| title             | Название мероприятия                                |
| description       | Описание                                            |
| starts_at         | Дата и время начала                                 |
| ends_at           | Дата и время окончания, опционально                 |
| location          | Место проведения, опционально                       |
| format            | Формат: `offline`, `online`, `hybrid`               |
| participant_limit | Лимит подтверждённых участников, опционально        |
| status            | Статус: `draft`, `active`, `cancelled`, `completed` |
| created_at        | Дата создания                                       |
| updated_at        | Дата обновления                                     |
| deleted_at        | Дата мягкого удаления, опционально                  |

Связи:

- `events.organizer_id` связан с `users.id`;
- один пользователь может быть организатором многих мероприятий;
- одно мероприятие имеет много записей участия.

### 3.3. event_participants

Хранит приглашения и статус участия пользователя в мероприятии. Организатор мероприятия определяется через `events.organizer_id` и не дублируется в этой таблице.

Поля:

| Поле         | Назначение                                                   |
| ------------ | ------------------------------------------------------------ |
| id           | Уникальный идентификатор записи                              |
| event_id     | Мероприятие                                                  |
| user_id      | Приглашённый пользователь                                    |
| status       | Статус участия: `invited`, `accepted`, `declined`, `removed` |
| invited_by   | Пользователь, который отправил приглашение                   |
| invited_at   | Дата приглашения                                             |
| responded_at | Дата принятия или отклонения                                 |
| removed_at   | Дата удаления из списка                                      |
| created_at   | Дата создания                                                |
| updated_at   | Дата обновления                                              |

Ограничения:

- пара `event_id` + `user_id` должна быть уникальной;
- приглашать можно только зарегистрированных пользователей;
- организатор не хранится как обычный участник в этой таблице;
- в ответе API списка участников организатор добавляется отдельным элементом с ролью `organizer`.

### 3.4. notifications

Хранит внутренние уведомления пользователя.

Поля:

| Поле             | Назначение                           |
| ---------------- | ------------------------------------ |
| id               | Уникальный идентификатор уведомления |
| user_id          | Получатель                           |
| type             | Тип уведомления                      |
| title            | Заголовок                            |
| message          | Текст                                |
| related_event_id | Связанное мероприятие, опционально   |
| read_at          | Дата прочтения, опционально          |
| created_at       | Дата создания                        |

Типы уведомлений:

- `event_invitation`;
- `event_updated`;
- `event_cancelled`;
- `participant_accepted`;
- `participant_declined`.

### 3.5. email_messages

Хранит email-сообщения, которые нужно отправить или уже отправлены пользователям.

Поля:

| Поле | Назначение |
| --- | --- |
| id | Уникальный идентификатор email-сообщения |
| user_id | Получатель, если это зарегистрированный пользователь |
| email | Email получателя |
| type | Тип письма |
| subject | Тема письма |
| body | Содержимое письма |
| status | Статус отправки: `pending`, `sent`, `failed` |
| related_event_id | Связанное мероприятие, опционально |
| sent_at | Дата успешной отправки, опционально |
| failed_at | Дата ошибки отправки, опционально |
| created_at | Дата создания |

Типы писем:

- `event_invitation`;
- `event_updated`;
- `event_cancelled`.

### 3.6. audit_logs

Опциональная таблица для действий администратора и важных изменений.

Поля:

| Поле        | Назначение                              |
| ----------- | --------------------------------------- |
| id          | Уникальный идентификатор записи         |
| actor_id    | Пользователь, который выполнил действие |
| action      | Тип действия                            |
| entity_type | Тип сущности                            |
| entity_id   | Идентификатор сущности                  |
| payload     | Краткие детали изменения                |
| created_at  | Дата действия                           |

На первом этапе таблицу можно не реализовывать, если нужна более простая версия проекта.

## 4. Связи между сущностями

Основные связи:

- `users 1 -> N events` через `events.organizer_id`;
- `users 1 -> N event_participants` через `event_participants.user_id`;
- `events 1 -> N event_participants` через `event_participants.event_id`;
- `users 1 -> N notifications` через `notifications.user_id`;
- `events 1 -> N notifications` через `notifications.related_event_id`;
- `users 1 -> N email_messages` через `email_messages.user_id`;
- `events 1 -> N email_messages` через `email_messages.related_event_id`.

Логические роли:

- гость: пользователь без авторизации;
- пользователь: запись в `users` с ролью `user`;
- администратор: запись в `users` с ролью `admin`;
- организатор: `events.organizer_id = current_user.id`;
- приглашённый: запись в `event_participants` со статусом `invited`;
- участник: запись в `event_participants` со статусом `accepted`.

Организатор отображается в списке участников как отдельная роль, но технически остаётся владельцем мероприятия через `events.organizer_id`.

## 5. Жизненные циклы

### 5.1. Статусы мероприятия

```text
draft -> active -> completed
             |
             v
         cancelled
```

Правила:

- новое мероприятие создаётся в статусе `draft`;
- приглашать участников можно только в `active`;
- редактировать основные данные может только организатор, пока мероприятие не `completed`;
- принять приглашение можно только для `active` мероприятия;
- отменённое и завершённое мероприятие остаётся доступным для просмотра;
- удаление лучше делать мягким через `deleted_at`, чтобы не ломать историю участия.

### 5.2. Статусы участия

```text
invited -> accepted
   |
   v
declined
```

Дополнительный переход:

```text
invited / accepted / declined -> removed
```

Правила:

- приглашение создаёт запись со статусом `invited`;
- принятие меняет статус на `accepted`;
- отклонение или отказ от участия меняет статус на `declined`;
- удаление организатором меняет статус на `removed`;
- повторное приглашение ранее удалённого или отказавшегося пользователя можно реализовать как смену статуса обратно на `invited`;
- отдельная таблица истории изменений статуса участия не создаётся.

| Метод  | URL                                               | Назначение                                   | Доступ       |

## 7. Правила доступа

Базовые проверки:

- гость может использовать только публичные страницы и auth-методы;
- заблокированный пользователь не может выполнять действия в закрытой части приложения;
- пользователь может редактировать только свой профиль;
- мероприятие может редактировать только его организатор;
- приглашать и удалять участников может только организатор активного мероприятия;
- приглашение может принять или отклонить только приглашённый пользователь;
- участник может отказаться только от своего участия;
- список организатора и подтверждённых участников видят организатор, приглашённые пользователи и участники;
- полный управленческий список со статусами видит только организатор;
- администратор может смотреть пользователей и мероприятия, а также блокировать пользователей и удалять мероприятия.

Для каждого запроса к мероприятию сервер должен вычислять отношение текущего пользователя к мероприятию:

- `isOrganizer`;
- `isInvited`;
- `isParticipant`;
- `isAdmin`.

На основе этих флагов API может возвращать список доступных действий для фронтенда.

## 8. Важные бизнес-правила

- Email пользователя должен быть уникальным.
- Приглашать можно только зарегистрированных пользователей.
- Нельзя пригласить пользователя в одно и то же мероприятие дважды как две разные записи.
- Нельзя открыть мероприятие пользователю, который не является организатором, приглашённым, участником или администратором.
- Нельзя принять приглашение на отменённое или завершённое мероприятие.
- Нельзя превысить лимит участников, если он задан.
- Организатор не может удалить сам себя из собственного мероприятия.
- Заблокированный пользователь не может создавать мероприятия, принимать приглашения или управлять участием.
- При удалении мероприятия оно должно исчезать из основных списков, но история может сохраняться через мягкое удаление.
- Администратор не становится организатором мероприятия, даже если может удалить его из админ-панели.
- На приглашение, изменение или отмену мероприятия создаётся внутреннее уведомление и email-сообщение.

## 9. Индексы и ограничения

Рекомендуемые ограничения:

- уникальный индекс на `users.email`;
- индекс на `events.organizer_id`;
- индекс на `events.status`;
- индекс на `events.starts_at`;
- уникальный индекс на `event_participants(event_id, user_id)`;
- индекс на `event_participants.user_id`;
- индекс на `event_participants.status`;
- индекс на `notifications.user_id`;
- индекс на `notifications.read_at`;
- индекс на `email_messages.user_id`;
- индекс на `email_messages.status`;
- индекс на `email_messages.related_event_id`.

Этого достаточно для базовых списков:

- мои мероприятия;
- мои приглашения;
- я участвую;
- участники мероприятия;
- административные списки.

## 10. Что можно отложить

Для первой версии можно не реализовывать:

- внешние email-приглашения для незарегистрированных пользователей;
- сложную систему уведомлений;
- повторяющиеся мероприятия;
- комментарии и чат мероприятия;
- загрузку изображений;
- сложный аудит всех изменений;
- интеграцию с календарями;
- публичные мероприятия без приглашений;
- отдельную историю изменения статусов участия.

## 11. Зафиксированные решения

- Приглашения отправляются только зарегистрированным пользователям.
- Организатор отображается в списке участников, но хранится через `events.organizer_id`.
- Все мероприятия закрытые: доступ есть только у организатора, приглашённых, участников и администратора.
- Email-уведомления входят в первую версию.
- Пользователи, имеющие доступ к мероприятию, могут видеть список участников.
- История изменения статусов участия не хранится отдельной таблицей.
